## 🎯 1. 화면 목적 (Purpose) - "왜 존재하는가?"

### 근본적인 이유

- **이 화면/기능은 왜 필요한가?**
- **이 화면이 없다면 사용자는 어떻게 할 것인가?**
- **앱의 전체 목표에서 이 화면의 역할은?**

### 한 문장 정의

- **이 화면을 한 문장으로 설명하면?**
  - "___을/를 위한 화면"
  - "사용자가 ___할 수 있는 곳"

### 우선순위

- **이 화면은 앱에서 얼마나 중요한가?** (핵심 / 중요 / 부가)
- **사용자의 여정에서 언제 등장하는가?** (초기 / 중간 / 후기)

---

## 👤 2. 사용자 목표 (User Goal) - "무엇을 달성하는가?"

### 주요 목표

- **사용자가 이 화면에서 달성하려는 것은?**
- **성공적으로 사용했다는 것은 어떤 상태인가?**
- **사용 후 사용자는 무엇을 얻는가?**

### 사용 시나리오

- **언제 이 화면을 방문하는가?**
- **어떤 상황에서 필요를 느끼는가?**
- **이전 화면에서 무엇을 하고 왔는가?**

### 사용자 감정

- **이 화면에 들어올 때 어떤 기분인가?** (궁금함? 답답함? 기대?)
- **나갈 때 어떤 기분이어야 하는가?** (만족? 성취감? 안심?)

---

## ⚡ 3. 핵심 액션 (Core Actions) - "무엇을 할 수 있는가?"

### 주요 행동

- **이 화면에서 할 수 있는 행동 Top 3는?**
  1. ___
  2. ___
  3. ___

- **가장 자주 하는 행동은?**
- **가장 중요한 행동은?** (빈도와 다를 수 있음)

### 액션 접근성

- **핵심 액션까지 몇 번의 탭/클릭?**
- **어디에 배치되어 있는가?** (상단? 하단? 플로팅 버튼?)
- **처음 방문한 사용자도 쉽게 찾을 수 있는가?**

### 보조 액션

- **필수는 아니지만 있으면 좋은 행동은?**
- **고급 사용자를 위한 숨겨진 기능은?**

---

## 📊 4. 정보 구조 (Information) - "무엇을 보여주는가?"

### 표시할 정보

- **이 화면에서 반드시 보여줘야 하는 정보는?**
- **정보의 우선순위는?** (가장 눈에 띄어야 하는 것부터)
  1. ___
  2. ___
  3. ___

### 정보 밀도

- **정보가 너무 많지 않은가?**
- **한눈에 파악 가능한가?**
- **스크롤이 필요한가?** (필요하다면 왜?)

### 빈 상태 (Empty State)

- **데이터가 없을 때 무엇을 보여주는가?**
- **사용자에게 어떤 행동을 유도하는가?**
- **왜 비어있는지 설명하는가?**

---

## 🔀 5. 화면 흐름 (Flow) - "어떻게 이동하는가?"

### 진입 경로

- **이 화면에 어떻게 들어오는가?**
  - 탭바에서?
  - 다른 화면에서 버튼 클릭?
  - 푸시 알림에서?
  - 딥링크에서?

### 퇴출 경로

- **이 화면에서 어디로 갈 수 있는가?**
- **뒤로가기 했을 때 어디로 가는가?**
- **작업 완료 후 어디로 보내는가?**

### 관련 화면

- **이 화면과 밀접한 다른 화면은?**
- **자주 왔다갔다 하는 화면은?**
- **함께 사용될 가능성이 높은 화면은?**

---

## 🔄 6. 상태 관리 (State) - "어떻게 변화하는가?"

### 화면 상태

- **이 화면의 가능한 상태는?**
  - 로딩 중
  - 데이터 있음
  - 데이터 없음
  - 에러 발생
  - 기타?

### 사용자 입력

- **사용자가 입력할 수 있는 것은?**
- **입력 검증이 필요한가?**
- **잘못된 입력 시 어떻게 하는가?**

### 데이터 동기화

- **서버와 동기화가 필요한가?**
- **오프라인에서 작동하는가?**
- **데이터 충돌 시 어떻게 처리하는가?**

---

## ✅ 7. 성공 지표 (Success) - "언제 성공인가?"

### 완료 조건

- **사용자가 "다 했다"고 느끼는 순간은?**
- **화면의 목적을 달성한 것은 어떻게 아는가?**

### 사용성 지표

- **이 화면에서 머무는 시간은?** (짧을수록 좋음? 길수록 좋음?)
- **작업 완료율은 얼마여야 하는가?**
- **에러 발생률은 얼마 이하여야 하는가?**

### 사용자 피드백

- **이 화면에 대해 사용자가 남기는 리뷰는?**
- **"이 화면 덕분에 ___했어요" - 빈칸에 들어갈 말은?**

---

## ⚙️ 8. 기술 구현 (Technical) - "어떻게 만드는가?"

### 데이터 소스

- **데이터를 어디서 가져오는가?**
  - 로컬 DB?
  - 서버 API?
  - 캐시?
  - UserDefaults?

### 의존성

- **이 화면이 의존하는 다른 모듈은?**
- **필요한 외부 라이브러리는?**
- **필요한 권한은?** (카메라, 위치, 알림 등)

### 성능 고려사항

- **대량의 데이터를 다루는가?**
- **이미지/비디오 처리가 필요한가?**
- **백그라운드 작업이 필요한가?**

### 재사용성

- **이 화면의 컴포넌트를 다른 곳에서 쓸 수 있는가?**
- **공통 UI 패턴을 따르는가?**
- **디자인 시스템과 일관성이 있는가?**

## 💡 예시: 클립키보드 - 템플릿 편집 화면

### 1. 개요
- **목적**: 사용자가 자주 쓰는 텍스트를 템플릿으로 저장하고 편집
- **우선순위**: 핵심 (앱의 가장 중요한 기능)
- **한줄 설명**: "반복 텍스트를 한번 저장하고 계속 재사용하는 화면"

### 2. 사용자 목표
- **달성하려는 것**: 매번 타이핑하는 텍스트를 템플릿으로 만들기
- **성공 조건**: 템플릿 저장 완료 → 키보드에서 바로 사용 가능
- **사용 시나리오**:
  - 자주 쓰는 이메일 서명 저장
  - 업무용 인사말 템플릿화
  - 주소, 전화번호 등 개인정보 저장

### 3. 핵심 액션
1. **텍스트 입력**: 저장할 내용 작성
2. **카테고리 선택**: 템플릿 분류
3. **저장**: 템플릿 DB에 저장

### 4. 화면 구성
- **필수 정보**:
  - 텍스트 입력 필드 (멀티라인)
  - 카테고리 선택 버튼
  - 저장/취소 버튼
- **보조 정보**:
  - 미리보기 (키보드에서 어떻게 보일지)
  - 글자 수 카운터
- **빈 상태**:
  - 플레이스홀더: "자주 사용하는 텍스트를 입력하세요"
  - 예시 템플릿 제안 (이메일 서명, 인사말 등)

### 5. 화면 흐름
- **진입**:
  - 템플릿 목록에서 "+" 버튼
  - 기존 템플릿 편집 시 탭
- **퇴출**:
  - 저장 완료 → 목록 화면으로 돌아가며 새 템플릿 하이라이트
  - 취소 → 목록 화면 (변경사항 없음)
- **관련 화면**:
  - 템플릿 목록 화면
  - 카테고리 관리 화면
  - 키보드 설정 화면

### 6. 상태 관리
- **가능한 상태**:
  - 신규 작성 모드
  - 편집 모드
  - 저장 중
- **입력 검증**:
  - 빈 텍스트는 저장 불가
  - 중복 템플릿 경고
- **에러 처리**:
  - 저장 실패 시 재시도 옵션
  - 너무 긴 텍스트 경고 (5000자 제한)

### 7. 성공 지표
- **완료 조건**: "저장" 버튼 탭 → 목록에 템플릿 추가됨
- **목표 완료율**: 90% 이상 (입력 시작한 사용자의 90%가 저장)
- **목표 사용 시간**: 평균 30초 이내 (빠르게 저장 가능해야 함)

### 8. 기술 구현
- **데이터 소스**:
  - Core Data (로컬 DB)
  - iCloud 동기화 (선택)
- **의존성**:
  - SwiftUI TextField (멀티라인)
  - Core Data Stack
- **성능 고려**:
  - 자동 저장 (5초마다 임시 저장)
  - 키보드 입력 시 실시간 미리보기

### 9. UI/UX 노트
- **디자인 원칙**:
  - 최소한의 UI 요소 (입력에 집중)
  - 저장 버튼은 항상 눈에 띄게
- **접근성**:
  - VoiceOver 지원
  - 다이나믹 타입 지원
- **다크모드**:
  - 입력 필드 배경 어둡게
  - 텍스트 대비 확보

## 🎨 화면 유형별 가이드

### 리스트 화면
- **목적**: 여러 항목을 탐색하고 선택
- **핵심 질문**:
  - 정렬 기준은?
  - 필터링 옵션은?
  - 무한 스크롤 vs 페이지네이션?
  - 항목 미리보기에 필요한 정보는?

### 상세 화면
- **목적**: 하나의 항목에 대한 모든 정보 표시
- **핵심 질문**:
  - 가장 중요한 정보는 무엇인가?
  - 편집/삭제 액션이 필요한가?
  - 관련 항목 표시가 필요한가?

### 입력 폼
- **목적**: 사용자로부터 정보 수집
- **핵심 질문**:
  - 필수 입력 필드는?
  - 입력 검증 시점은? (실시간 vs 제출 시)
  - 자동 완성/제안이 필요한가?
  - 임시 저장이 필요한가?

### 설정 화면
- **목적**: 앱 동작 커스터마이징
- **핵심 질문**:
  - 그룹핑 기준은?
  - 기본값은 무엇인가?
  - 변경 즉시 적용 vs 저장 버튼?
  - 초기화 옵션이 필요한가?

### 대시보드
- **목적**: 핵심 지표를 한눈에 파악
- **핵심 질문**:
  - 표시할 지표는 최대 몇 개?
  - 실시간 업데이트가 필요한가?
  - 상세 화면으로의 드릴다운이 필요한가?

## 💭 설계 원칙

### 1. 간결함 (Simplicity)
- 한 화면 = 하나의 주요 목적
- 핵심 액션은 3개 이하
- 정보는 최소한으로

### 2. 명확함 (Clarity)
- 사용자가 무엇을 할 수 있는지 명확
- 피드백은 즉각적
- 에러 메시지는 구체적

### 3. 일관성 (Consistency)
- 같은 액션은 같은 방식으로
- UI 패턴 재사용
- 용어 통일

### 4. 효율성 (Efficiency)
- 최소한의 탭/클릭
- 자주 하는 행동은 쉽게
- 반복 작업은 자동화

### 5. 용서 (Forgiveness)
- 실수 복구 가능
- 되돌리기 지원
- 위험한 액션은 확인 필요
